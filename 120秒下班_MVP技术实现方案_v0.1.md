# 《120秒下班》MVP 技术实现方案 v0.1

## 1. 技术目标

本方案服务于 MVP v0.1，目标不是一次性做成完整商业版本，而是快速做出可试玩、可调数、可扩展到微信小游戏的核心闭环。

核心目标：

| 目标 | 说明 |
|---|---|
| 快速可玩 | 优先在 PC Web 跑通完整 120 秒单局 |
| 移动优先 | 画面、UI、输入按竖屏手机设计 |
| 低模 3D 表现 | 使用 3D 场景、低模办公室、固定斜俯视相机建立地图形式 |
| 配置驱动 | 玩家、敌人、武器、弹药、技能、道具、时间线、地图点位尽量配置化 |
| 模块清晰 | 输入、武器、战斗、弹药、技能、道具、UI、结算边界明确 |
| 可移植 | 玩法逻辑不强绑定 DOM 或浏览器事件，便于后续适配微信小游戏 |

---

## 2. 推荐技术栈

### 2.1 MVP 推荐方案

```text
TypeScript
Vite
Three.js
WebGL 渲染
HTML/CSS UI 覆盖层
JSON/TS 配置表
```

选择原因：

| 技术 | 作用 | 原因 |
|---|---|---|
| TypeScript | 玩法逻辑开发 | 类型约束能降低配置和实体字段错误 |
| Vite | Web 开发与构建 | 启动快，适合快速调试 |
| Three.js | 3D 渲染和场景管理 | 适合低模办公室、斜俯视相机、灯光和阴影 |
| WebGL | 渲染层 | 支持 3D 场景、材质、光照和阴影 |
| HTML/CSS | UI 层 | HUD、摇杆、按钮、升级面板用 DOM 更高效清晰 |
| TS 配置表 | 数值配置 | 开发期类型更清晰，后续可转 JSON |

### 2.2 微信小游戏适配策略

MVP 第一阶段先完成 PC Web 可试玩版本。微信小游戏适配作为后续平台层任务处理。

适配原则：

```text
玩法逻辑不直接读取 window/document
输入模块统一输出 InputState
音频、存档、生命周期放到 PlatformAdapter
资源路径集中管理
画布尺寸由统一的 ScaleManager 控制
```

如果后续要求直接以微信小游戏为主发布，需要在 M8/M10 阶段评估：

| 方案 | 适用情况 | 代价 |
|---|---|---|
| Three.js Web 构建 + 微信适配层 | 已经完成 Web MVP，希望低成本迁移 | 需要处理 WebGL 适配、包体、触屏和音频限制 |
| Cocos Creator 3D 重建项目 | 微信小游戏发布优先级高，后续 3D 内容量大 | 初期开发和工程管理成本更高 |

当前 MVP 建议先采用 Three.js Web 方案，尽快验证 3D 低模地图表现和玩法闭环。

## 4. 核心架构

### 4.1 游戏状态

使用明确的单局状态机管理流程：

```text
Ready
  ↓
Playing
  ↓
LevelUpPaused
  ↓
Playing
  ↓
Success / Failed
  ↓
Restart
```

状态定义：

| 状态 | 说明 |
|---|---|
| Ready | 场景加载完成，等待开局 |
| Playing | 正常游戏中，倒计时、敌人、技能全部更新 |
| LevelUpPaused | 升级选择中，玩法暂停，UI 可交互 |
| Success | 成功撤离结算 |
| Failed | 死亡或超时失败结算 |

### 4.2 每帧更新顺序

每帧按固定顺序更新，减少系统之间互相踩状态。

```text
读取输入
  ↓
更新时间线
  ↓
更新玩家移动与状态
  ↓
刷新和更新敌人
  ↓
更新武器、攻击触发与短生命周期射击表现
  ↓
处理碰撞和伤害
  ↓
处理道具拾取
  ↓
处理经验和升级
  ↓
处理撤离读条
  ↓
检查成功/失败
  ↓
刷新 UI
```

`LevelUpPaused` 状态下只更新升级 UI，不更新倒计时、敌人、技能和碰撞。

## 5. 输入系统

### 5.1 统一输入结构

键盘、可选指针、单摇杆、射击按钮、换弹按钮和技能按钮统一转换成同一个输入对象。

```ts
export type InputState = {
  moveX: number;
  moveZ: number;
  aimX: number;
  aimZ: number;
  fireHeld: boolean;
  reloadPressed: boolean;
  dashPressed: boolean;
};
```

约束：

| 字段 | 说明 |
|---|---|
| moveX / moveZ | 范围 -1 到 1，内部做归一化，作用于地面 x/z 平面 |
| aimX / aimZ | 默认取归一化后的移动方向；停止移动后保留最后有效方向 |
| fireHeld | 只表示玩家当前是否主动要求开火，不直接生成伤害 |
| reloadPressed | 只在按下帧为 true，交给 WeaponSystem 校验 |
| dashPressed | 只表示本帧是否触发主动技能 |

### 5.2 PC 输入

| 按键 | 输出 |
|---|---|
| W / ↑ | moveZ = -1 |
| S / ↓ | moveZ = 1 |
| A / ← | moveX = -1 |
| D / → | moveX = 1 |
| J / 鼠标左键按住 | fireHeld = true |
| 触控板或鼠标移动 | 角色停止时，从屏幕射线与地面交点计算 aimX / aimZ |
| R | reloadPressed = true |
| Space | dashPressed = true |

### 5.3 手机输入

| 控件 | 输出 |
|---|---|
| 左下角虚拟摇杆 | moveX / moveZ，并同步生成 aimX / aimZ |
| 右下角射击按钮 | 按住时 fireHeld = true |
| 换弹按钮 | reloadPressed |
| 射击按钮上方技能按钮 | dashPressed |

虚拟摇杆和按钮只影响输入状态，不直接移动玩家或生成攻击。

瞄准优先级：

```text
存在移动输入：移动方向覆盖 aimDirection
没有移动输入且指针本帧移动：指针地面交点更新 aimDirection
两者都没有：保留最后有效 aimDirection
```

## 6. 地图与坐标

### 6.1 坐标系统

采用 3D 世界坐标、玩法平面坐标和屏幕坐标分离：

| 坐标 | 用途 |
|---|---|
| 3D 世界坐标 | Three.js 场景中的 mesh 位置，使用 x/y/z |
| 玩法平面坐标 | 玩家、敌人、道具、撤离区和碰撞，使用 x/z |
| 屏幕坐标 | DOM UI、摇杆、按钮、提示 |

约定：

```text
x：地图横向
z：地图纵向
y：高度
```

摄像机使用固定斜俯视角跟随玩家，UI 固定在屏幕坐标。

### 6.2 地图配置

地图先用矩形区域配置玩法范围，再由 Three.js 低模几何体渲染地面、墙体和家具。

```ts
export type MapArea = {
  id: string;
  name: string;
  x: number;
  z: number;
  width: number;
  depth: number;
};
```

关键点位：

```text
playerSpawn
accessCardSpawn
coffeeSpawn
ammoSpawnWorkstation
ammoSpawnBossOffice
bossSpawn
elevatorZone
```

### 6.3 障碍物

MVP 第一版只使用少量盒体障碍：

```text
办公桌
会议桌
咖啡机
老板桌
电梯门装饰
```

碰撞策略：

| 对象 | 策略 |
|---|---|
| 玩家 vs 障碍 | 在 x/z 平面做 AABB 或圆形近似阻挡 |
| 敌人 vs 障碍 | 使用共享流场寻路和圆形滑动碰撞 |
| 枪械射线 vs 障碍 | 射线只命中最近交点，墙体和硬家具优先阻挡后方敌人 |

### 6.4 相机

MVP 使用 `OrthographicCamera`，固定斜俯视角，跟随玩家目标点平滑移动。

```text
相机类型：正交相机
观察角度：斜俯视
目标点：玩家 x/z 位置
UI：不进入 3D 场景，使用 DOM 覆盖层
```

## 8. 敌人与刷怪

### 8.1 刷怪阶段

刷怪由时间线控制，不直接写死在敌人模块中。

| 时间 | 刷怪内容 |
|---|---|
| 0 秒 | Bug 怪开始刷新 |
| 15 秒 | 需求变更怪加入刷新池 |
| 40 秒 | 会议怪加入刷新池 |
| 90 秒 | 老板出现 |

### 8.2 刷怪位置

普通敌人从屏幕外或地图边缘生成，优先保证玩家能看到敌人涌入。

规则：

```text
根据玩家当前位置选择视野外方向
生成点限制在地图边界内
避免生成在玩家半径过近区域
拒绝墙体、家具和门洞通行区域中的候选点
```

### 8.3 敌人行为

MVP 使用静态障碍网格和共享流场追击：

```text
障碍变化或玩家跨网格时更新流场
敌人读取相邻网格中更低的距离值
近距离且视线无遮挡时直接追击
移动使用圆形分轴碰撞并叠加敌人分离力
接触后造成伤害
```

会议怪额外效果：

```text
接触玩家后施加 30% 减速，持续 1 秒
```

老板额外效果：

```text
第 90 秒出现
持续追击玩家
接触高伤害
P1 阶段周期性召唤需求变更怪
```

## 9. 武器、攻击与技能系统

### 9.1 统一攻击接口

攻击触发方式与伤害执行分离。装备先根据 `attackMode` 产生攻击请求，再由 `CombatSystem` 统一处理射线、阻挡、穿透、伤害和反馈。

```ts
export type AttackRequest = {
  sourceId: string;
  weaponId: string;
  mode: "manual" | "automatic";
  originX: number;
  originZ: number;
  directionX: number;
  directionZ: number;
};
```

```text
manual：读取玩家瞄准方向和 fireHeld
automatic：由装备按冷却和选敌规则生成方向
两种模式：统一进入 CombatSystem
```

MVP 默认冲锋枪只使用 `manual`。`automatic` 接口保留并测试空实现，不配置具体自动攻击装备。

### 9.2 默认冲锋枪状态机

```text
Ready
  ├─ fireHeld 且有弹药、冷却结束 -> Fire -> Ready
  ├─ reloadPressed 且弹匣未满、后备弹药大于 0 -> Reloading
  └─ 弹匣为空 -> Reloading

Reloading
  ├─ 时间结束 -> 从后备弹药补足弹匣 -> Ready
  └─ 咖啡冲刺 -> 取消换弹 -> Ready
```

换弹期间允许移动但不能射击。战术换弹只补充弹匣缺口，不丢弃弹匣剩余子弹，也不生成免费弹药。

### 9.3 射线命中

每次合法射击执行一次玩法平面射线检测：

```text
从枪口沿 aimDirection 建立射线
收集射程内敌人圆形碰撞体和障碍 AABB 的交点
按距离从近到远排序
硬障碍终止射线
敌人命中后结算伤害和穿透衰减
生成短生命周期曳光线和命中特效
```

射线判定与曳光线表现分离，不创建长期存在的实体子弹。穿透升级只能穿透敌人，不能穿透墙体和硬家具。

### 9.4 枪械升级

升级系统保留 3 选 1，枪械强化分为：

```text
火力校准：伤害、射速、散布、暴击
弹匣管理：容量、换弹时间、换弹后增益
加班穿透弹：敌人穿透、衰减、精英伤害和击退
```

升级只修改配置派生值，不直接修改输入模式，也不能把默认枪械改成自动索敌武器。

### 9.5 咖啡冲刺

实现方式：

```text
读取当前移动方向
如果没有移动方向，则使用最近一次有效方向
短时间提升位移速度或直接位移一段距离
触发冷却
```

配置字段：

```text
cooldown
distance
duration
knockback
postSpeedBoost
charges
```

### 9.6 自动攻击扩展约束

后续自动攻击副装备复用 `AttackRequest` 和伤害管线，但自行负责目标选择、触发冷却和表现实体。自动装备不得消耗默认枪械弹匣，是否使用独立弹药由装备配置决定。

## 10. 道具系统

### 10.1 门禁卡

门禁卡由时间事件在第 35 秒生成。

拾取后：

```text
player.hasAccessCard = true
隐藏门禁卡实体
更新 HUD
切换目标箭头逻辑
```

### 10.2 咖啡

咖啡在茶水间固定刷新。

拾取后：

```text
回复 20 血
施加 20% 移速加成，持续 3 秒
进入刷新冷却
```

### 10.3 弹药补给

固定弹药箱在工位区和老板办公室刷新，小型弹药包由普通敌人概率掉落。

拾取后：

```text
reserveAmmo = min(reserveAmmoMax, reserveAmmo + supplyAmount)
如果后备弹药已满，则不消耗补给
固定补给按刷新冷却重新出现
```

固定补给保证玩家可以规划路线，随机掉落只用于降低中途断粮压力。

### 10.4 离职信

离职信可以从敌人小概率掉落，或先在固定点刷新。

拾取后：

```text
立即释放一次范围伤害
普通敌人高伤害
老板低倍率伤害
```

## 12. 撤离系统

撤离条件：

```text
玩家持有门禁卡
电梯已开放
玩家在电梯区域内
玩家未死亡
```

读条规则：

```text
进入电梯区域后开始累计
离开区域后清零
累计达到 2 秒触发成功结算
读条期间仍然会受到伤害
```

失败优先级：

```text
如果读条期间死亡，进入死亡失败
如果时间归零且未成功，进入超时失败
如果同一帧读条完成，成功优先于超时
```

## 16. 性能策略

MVP 阶段重点控制敌人、射线检测和短生命周期射击特效数量。

策略：

| 项目 | 策略 |
|---|---|
| 敌人数量 | 设置全局上限 |
| 射击表现 | 曳光线和命中粒子对象池复用 |
| 伤害检测 | 敌人较少时线性射线检测，超过预算后做空间分桶 |
| UI 更新 | 数据变化时更新，避免每帧重建 DOM |
| 特效 | MVP 使用简单图形和短生命周期粒子 |

建议上限：

```text
全部敌人（含老板）：30
同时存在的曳光线：24
预留自动装备投射物：40
掉落道具：20
浮动伤害数字：30
```

## 19. 风险与处理

| 风险 | 表现 | 处理 |
|---|---|---|
| 枪械系统改造范围过大 | 输入、战斗和 UI 同时耦合导致返工 | 先拆 Input/Weapon/Combat，再接射击表现和升级 |
| 弹药耗尽导致战斗停滞 | 玩家无法获得经验或突破包围 | 至少两个固定补给点，随机掉落只作补充 |
| 刷怪过多导致卡顿 | 90 秒后帧率下降 | 敌人数量上限、对象池、简化碰撞 |
| 手机操作卡手 | 摇杆和射击按钮重叠或角色被障碍卡住 | 固定控制区尺寸、扩大通道、验证常见竖屏尺寸 |
| 目标不清晰 | 玩家不知道去哪里 | 时间提示 + 目标箭头 + 状态 UI |
| 微信适配返工 | Web 逻辑依赖浏览器 API | 平台能力统一封装到 PlatformAdapter |
